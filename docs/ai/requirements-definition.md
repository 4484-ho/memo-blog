## 明確に決めておく必要があること

### 要件面
- どのModelを使用するのか
  - コスト・特性のほか、リージョンやデータの保存先、学習利用等の条件を精査する必要がある
  - 扱えるコンテキストのサイズも検討対象となる
  - **レイテンシ（応答速度）の要件**: モデルのサイズや推論速度がユーザー体験に直結するため、許容できる待機時間を定義する
- どのような推測をさせたいのか
  - AIがまとめる、程度では足りない。期待する挙動を言語化する必要がある。
    - どんなInputに対し、
    - 何を判断させ
    - どんなOutputを得たいのか 
  - ポイントとしては、専門家に何を依頼するのか、どんな回答を期待するのかという話と同じと思っている
- **評価指標（Evaluation）をどうするか**
  - AIの出力が「期待通り」であることをどう測定するか（精度、網羅性、ハルシネーションの少なさなど）
  - 定量的な評価（LLM-as-a-Judgeなど）と定性的な人間の評価（Human-in-the-loop）の仕組み
  - ドッグフーディング含む調整の期間を必ず必要とする
- **セキュリティとガバナンス**
  - PII（個人情報）や機密情報のマスキング・フィルタリング方針
  - プロンプトインジェクションやジェイルブレイクといったAI特有の脆弱性への対策
- **ハルシネーションを前提としたUX/UI設計**
  - AIは間違える可能性があることをユーザーにどう提示するか
  - ユーザーが出力結果を修正・破棄・フィードバックできるUIの検討


### 技術面

- どのようなフレームワークを使用するのか
  - LangChainだけではない
    - LangGraph (LangChainエコシステムでAgent構築に特化)
    - LlamaIndex (データ接続・RAGに強くAgent機能も持つ)
    - AutoGen (Microsoft製のマルチエージェントフレームワーク)
    - CrewAI (役割を持った複数のエージェントを連携させるフレームワーク)
    - Semantic Kernel (Microsoft製のエンタープライズ向けフレームワーク)
    - Dify (ローコード/ノーコードでAgentを構築できるプラットフォーム)
  - 適切なフレームワークを選定することが重要
- どのようなWorkflowを組むのか
  - RAGの利用や、Agenticなワークフローなどの選定は前述の「どのような推測をさせたいのか」の言語化を行ってから検討すること
  - 期待するAgent,AIの挙動が言語化されて、適切なWorkflowを組むことができる。
  - 一方で、どんなツールを使用してどのような順序やReActな動作の期待やハーネスを考える必要がある
- **コンテキスト・メモリ管理**
  - 過去のやり取りや状態をどこまで保持するか（セッション単位か、永続化するか）
  - トークン制限に収めるための要約やベクトル検索による関連記憶の引き出し方
- タイムアウトにならない対策
  - LLM側の処理が長時間化する傾向にある。
    - Streamやポーリング等も含め、タイムアウトで処理がエラーとなるようなことがないよう、設計が必要
    - **非同期処理（バックグラウンドジョブ）とWebSocket/Webhook等による完了通知のアーキテクチャ**も検討する
- Fallbackについて
  - エラーが発生した場合のFallback処理を考える必要がある
    - 処理だけでなく、Modelやプラットフォーム、ベンダー自体を切り替えるような構想も必要となる
    - APIのレートリミット（Rate Limit）に到達した場合のリトライ戦略（Exponential Backoffなど）
- **プロンプト管理とテスト（LLMOps）**
  - プロンプトのバージョン管理をどう行うか（コードとして管理するか、外部のCMS/プロンプト管理ツールを使うか）
  - **監視・オブザーバビリティ**: トークン消費量、レイテンシ、ユーザーの入力とAIの出力をモニタリングする仕組み（LangSmith, Phoenixなど）
